# Stone IM Binary Protocol

This document defines the semantic contract carried by the binary node codec in
`src/stone_protocol`.

## Frame

Each WebSocket binary message or TCP stream frame uses:

```text
0               1               2
+---------------+---------------+---------------+
| flag:1 |             length:23                 |
+---------------+---------------+---------------+
| payload: length bytes                          |
```

- `flag = 0` means a plain Stone protocol payload.
- `flag = 1` is reserved and must currently be rejected.
- `length` is the encoded node payload length and must be no larger than
  `0x7fffff`.

The payload is a single encoded node. Decoders may return trailing bytes to
support stream parsing, but a single frame must not contain multiple root nodes.

## Node Model

A node is:

```erlang
{Desc :: binary(), Attrs :: [{binary(), binary() | {jid, binary()}}], Content}
```

`Content` is one of:

- `nil`
- `binary()`
- `[Node]`

Node names and attributes may use token compression. Binary content is encoded
as binary data even when it equals a token string.

## Connection State

The protocol state machine is:

```text
connected -> negotiated -> authenticated -> closing
```

Allowed nodes by state:

- `connected`: client may send `hello`; server may send `hello_ack` or `error`.
- `negotiated`: client may send `auth`, `ping`; server may send `auth_ok`,
  `error`, `pong`.
- `authenticated`: both sides may exchange `message`, `ack`, `nack`, `retry`,
  `receipt`, `presence`, `offline_pull`, `offline_batch`, `ping`, `pong`,
  `error`.
- `closing`: only best-effort `error` or transport close is expected.

Any state violation must produce `error` code `state_violation` and the server
may close the connection.

## Version and Capabilities

Client starts with:

```erlang
stone_message:hello(DeviceId, ClientVersion, Capabilities)
```

Server replies:

```erlang
stone_message:hello_ack(SessionId, SelectedVersion, EnabledCapabilities)
```

Rules:

- Current protocol version is `<<"1">>`.
- `hello` carries `min_version` and `max_version`; the server selects exactly
  one version from that inclusive range.
- This implementation currently accepts only version `1`.
- Capabilities are represented as repeated `cap` child nodes with a `name`
  attribute.
- Unknown capabilities are ignored unless the node requires them.

Initial capability names:

- `ack`: durable message acknowledgement.
- `retry`: explicit retry request.
- `offline`: offline message pull and batch delivery.
- `receipt`: read receipts.
- `presence`: user presence.

## Authentication

After negotiation, the client sends:

```erlang
stone_message:auth(SessionId, UserId, DeviceId, Token)
```

Server replies:

```erlang
stone_message:auth_ok(UserId, LastSeq)
```

or:

```erlang
stone_message:auth_fail(Code, AuthId, Detail)
```

Rules:

- `SessionId` must match the `hello_ack` session.
- `UserId` identifies the authenticated account.
- `DeviceId` identifies the client device and must remain stable for retry and
  offline sequencing.
- `LastSeq` is the highest contiguous server-to-device sequence durably accepted
  by the server for that device.
- No user data nodes are accepted before `auth_ok`.

## Message Identity and Deduplication

Every user message has an `id` generated by the sender. Servers must treat
`{from, device, id}` as the idempotency key for client-to-server messages.

When the server accepts a message, it assigns a monotonically increasing `seq`
per recipient device. Server-to-client de-duplication uses `{to, device, seq}`.

Rules:

- Re-sending the same `{from, device, id}` must not create a duplicate stored
  message.
- Re-delivering the same `{to, device, seq}` must not create a duplicate on the
  client.
- Clients must ack the highest contiguous sequence they have durably stored.

## Ack, Nack and Retry

Successful durable acceptance:

```erlang
stone_message:ack(Id, To, Seq)
```

Negative acknowledgement:

```erlang
stone_message:nack(Id, To, Code, Detail)
```

Explicit retry request:

```erlang
stone_message:retry(Id, To, Attempt)
```

Rules:

- `ack` is sent only after the receiver has durably stored or applied the node.
- `nack` uses the shared error code list below.
- Retry attempts should use exponential backoff with jitter.
- The same message `id` must be reused during retry.
- A sender should stop retrying on non-retryable error codes.

## Offline Delivery

Client requests missed messages:

```erlang
stone_message:offline_pull(AfterSeq, Limit)
```

Server replies:

```erlang
stone_message:offline_batch(More, Messages)
```

Rules:

- `AfterSeq` is the last contiguous server-to-device sequence the client has
  durably stored.
- `Limit` caps the number of messages in one batch.
- Each offline `message` must include `seq`.
- `More = <<"true">>` means the client should issue another `offline_pull`.
- `More = <<"false">>` means the device is caught up at the time of the batch.
- The client must still ack delivered offline messages.

## Error Codes

Errors use:

```erlang
stone_message:error(Code, Id, Detail)
```

Standard codes:

| Code | Retry | Meaning |
| --- | --- | --- |
| `bad_frame` | no | Frame header or payload cannot be decoded. |
| `unsupported_flag` | no | Frame flag is reserved or unsupported. |
| `unsupported_version` | no | No compatible protocol version exists. |
| `state_violation` | no | Node is not allowed in the current connection state. |
| `auth_required` | no | User data arrived before authentication. |
| `auth_failed` | no | Token or session is invalid. |
| `invalid_jid` | no | JID cannot be encoded or routed. |
| `duplicate` | no | Message was already accepted. The previous ack should be reused. |
| `rate_limited` | yes | Sender exceeded allowed rate. |
| `unavailable` | yes | Recipient route or storage is temporarily unavailable. |
| `internal` | yes | Server failed before a durable decision was made. |

When an error is associated with a message, `Id` must be that message id.
Connection-level errors use `Id = <<"">>`.

## Decoder Limits

Decoders must reject inputs that exceed local resource limits before allocating
large structures. The default limits are:

| Limit | Default |
| --- | --- |
| `max_depth` | `32` nested node levels |
| `max_attrs` | `64` attributes per node |
| `max_children` | `256` child nodes per content list |
| `max_binary_size` | `1048576` bytes per binary value |

Limit failures are surfaced as `{limit_exceeded, LimitName}` and should be
reported to peers as `bad_frame`.
